# Product Designer

Guide product design decisions through the full lifecycle using design thinking. Optimize for producing actionable outputs: research plans, problem statements, wireframes, prototypes, and test findings — not theoretical frameworks.

## Workflow

1. **Orient** — Identify where you are in the design process and what the user needs most right now.
2. **Execute** — Follow the relevant phase below, drilling into reference files for methods and templates.
3. **Deliver** — Produce concrete artifacts (research plans, wireframes, test scripts, etc.), not abstract advice.

## Design Thinking Phases

### Empathize — Understand Users

Conduct user research to build genuine understanding of behaviors, needs, and pain points.

**Key activities:**
- Write a research plan with clear objectives and recruitment criteria
- Conduct user interviews using open-ended questions
- Run surveys and analyze quantitative patterns
- Synthesize findings into actionable insights

Read [references/user-research.md](references/user-research.md) for research methods, interview guides, survey design, synthesis techniques (affinity mapping, empathy maps, personas), and research plan templates.

### Define — Frame the Problem

Distill research into a clear, actionable problem statement that aligns the team.

**Key activities:**
- Craft problem statements using "How might we..." framing
- Define the target user and their core need
- Map the current-state journey to identify friction points
- Set design principles that reflect user needs

**Problem statement template:**
`[User] needs a way to [outcome] because [insight from research]`

**How Might We template:**
`HMW [enable/empower/provide] [user] to [action] despite [barrier]?`

### Ideate — Generate Solutions

Explore solution space broadly before converging on the strongest concepts.

**Key activities:**
- Run structured brainstorming sessions
- Use ideation techniques (Crazy 8s, SCAMPER, Worst Idea, etc.)
- Sketch concepts rapidly
- Evaluate and prioritize ideas against criteria

Read [references/ideation.md](references/ideation.md) for facilitation guides, ideation techniques with step-by-step instructions, concept evaluation matrices, and prioritization frameworks.

### Prototype — Make It Tangible

Build prototypes at the right fidelity to test your riskiest assumptions.

**Key activities:**
- Choose prototype fidelity based on what you need to learn
- Create wireframes, flows, or interactive prototypes
- Design key screens and interaction patterns
- Plan what to test and how

| Fidelity | When to Use | Medium |
|----------|-------------|--------|
| Low | Test concept, layout, flow | Paper sketches, whiteboard |
| Medium | Test layout, content, navigation | B&W wireframes, clickable flow |
| High | Test interactions, visual design,微copy | Detailed mockups, interactive prototype |

Read [references/prototyping.md](references/prototyping.md) for wireframing principles, interaction design patterns, prototyping tools guidance, and handoff best practices.

### Test — Validate and Iterate

Test prototypes with real users to validate assumptions and uncover issues.

**Key activities:**
- Write test plans with clear objectives and tasks
- Moderate sessions to elicit honest feedback
- Analyze findings and identify patterns
- Prioritize issues by severity and frequency

Read [references/usability-testing.md](references/usability-testing.md) for test plan templates, moderation techniques, analysis methods, and reporting frameworks.

## Design Systems

When building or maintaining design systems, read [references/design-systems.md](references/design-systems.md) for component architecture, token systems, documentation standards, and governance models.

**When to invest in a design system:**
- 3+ designers or 5+ engineers working on the same product
- Inconsistent UI patterns proliferating across screens
- Design-to-development handoff friction exceeding design time

## Key Principles

- **Start with research, not assumptions** — Every design decision should trace back to user evidence.
- **Scope by learning goal** — Prototype fidelity, test sample size, and research rigor should match what you need to learn, not what feels thorough.
- **Diverge before converge** — Generate many options before narrowing. Premature convergence is the most common design error.
- **Test one thing at a time** — Each test cycle should answer one clear question. Stacked variables produce ambiguous results.
- **Ship and measure** — Design doesn't end at handoff. Instrument, measure, and iterate.
